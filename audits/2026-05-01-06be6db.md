## Findings by Severity

### CRITICAL

#### Reliability ‚Äî `EventLog.load()` crashes on a single malformed JSONL line (REL-001)
- **Where:** `godel/_event_log.py:190`
- **Evidence:**
  ```python
  with open(file_path) as f:
      for line in f:
          line = line.strip()
          if not line: continue
          d = json.loads(line)        # no try/except
          event = Event.from_dict(d)
  ```
- **Why it matters:** Replay depends on `EventLog.load`. The writer `flush()`'s but never `fsync()`'s (REL-002), so any SIGKILL/OOM/power-loss can leave a torn final line. `godel resume <run_id>` then hits `JSONDecodeError` and aborts before any prior events are consumed ‚Äî the entire audit trail is unrecoverable through the normal API even though 99.99% is intact. By contrast `_run_summary.py:_load_events_readonly` already wraps `json.loads` in try/except ‚Äî the asymmetry confirms the issue is known elsewhere.
- **Fix:** Wrap `json.loads` + `Event.from_dict` in try/except `(JSONDecodeError, KeyError, TypeError, ValueError)`; log warning with `run_id` and offending byte-offset; truncate at the bad line. Match `_run_summary.py`'s lenient policy.
- **Effort:** S

#### Reliability ‚Äî `EventLog._append_event` flushes but never fsyncs (REL-002)
- **Where:** `godel/_event_log.py:157`
- **Evidence:**
  ```python
  line = json.dumps(event.to_dict(), separators=(",", ":"))
  self._file.write(line + "\n")
  self._file.flush()       # userspace flush only ‚Äî no os.fsync
  ```
- **Why it matters:** The audit log is THE source of truth for replay. `flush()` only pushes bytes from Python buffers to the kernel; an OS crash or container OOM-kill between `flush()` and kernel writeback can lose seconds of events the process believes durable. `_transcript.py:340` and `io.py:407` both explicitly `fsync` because they rely on durability ‚Äî the event log, more critical, does not. After such a crash, the log can be missing FINISHED events that the replay walker treats as STARTED-only, breaking the exactly-once guarantee for non-idempotent steps.
- **Fix:** Add `os.fsync(self._file.fileno())` after `flush()`. Wrap in `try/except OSError` (some FS/sandboxes refuse fsync) mirroring `_transcript.py:_flush_and_sync`. Default fsync-per-event; Godel is correctness-bound, not throughput-bound.
- **Effort:** S

#### Reliability ‚Äî `EventLog` has no lock; `parallel()` branches race on `_seq_counter` (REL-003)
- **Where:** `godel/_event_log.py:69`
- **Evidence:**
  ```python
  # emit_started:
  event = Event(... seq=self._seq_counter, ...)
  self._seq_counter += 1                # non-atomic RMW
  # _decorators.py:894 ‚Äî branches share ctx.event_log:
  results = await asyncio.gather(
      *[_run_branch(aw) for aw in aws], return_exceptions=True
  )
  ```
- **Why it matters:** `parallel()` shares `EventLog` across branch coroutines. Pure-async on a single loop is mostly safe, BUT (a) `capture_stdout=True` spawns a reader **thread** in `_stdout_capture.py:142` that calls into transcript/event paths; (b) the watcher subprocess and producer threads in `_watch.py` read this file concurrently; (c) any `await` inside the call chain (parent re-persist on line 78 ‚Üí `_append_event` ‚Üí can block on flush during disk pressure) creates a non-atomic window where two branches read the same `_seq_counter`. Result: duplicate seq numbers in the JSONL ‚Äî replay walker `try_match`'s positional index becomes ambiguous and a step's response can be hijacked by its sibling's cache. `_transcript.py` already uses `threading.Lock()` (line 135) for exactly this reason.
- **Fix:** Add `threading.Lock` (not `asyncio.Lock` ‚Äî writer is sometimes called from a reader thread) in `EventLog.__init__`; acquire around counter increment + write in `emit_started`/`emit_finished`/`emit_failed`/`emit_suspended`/`_append_event`.
- **Effort:** M
